## What it does

Documents how the URL-extraction pipeline captures **interaction-state styles** in addition to the resting computed style. Variant 1 DESIGN.md has no `:hover` / `:focus` slot in the canonical schema, so the extracted state styles are written to the **extension namespace** `extensions.states` per [extension-sections-10-14](../../amw-design-md-spec/references/extension-sections-10-14.md). A site whose hover state shifts a button from `#5e6ad2` → `#6f7be0` exposes brand information that a one-pass resting-state extractor misses entirely.

State extraction runs on demand via the `--extract-states` flag of `bin/amw-design-md-from-url.sh`. Without that flag the extractor stays in its default fast path (resting state only). The flag enables a second probing pass that doubles to triples extraction time but captures the missing 30-40% of the design system that lives only in pseudo-classes.

## When to use

- User says: "extract DESIGN.md including hover states"
- User says: "I need the full interaction palette of <site>"
- Brand-researcher: extracting from a site where micro-interactions are part of the brand (Linear, Vercel, Stripe — sites that ship deliberate hover/focus systems)
- The extracted resting-state DESIGN.md fails the contrast check on button text against button background (often a hover/focus alternate fixes the contrast)

## When NOT to use

- User explicitly only wants Variant 1 core tokens (resting only) — skip; do not pollute frontmatter with extension namespace
- Site disables hover on touch (`@media (hover: hover)` guards) — extraction will not trigger them; record `warnings: ["site gates hover behind @media (hover); pointer simulation has no effect"]`
- Single-page extraction budget is tight (<5 seconds per page) — state probing adds ~200ms per candidate × 10-20 candidates = 2-4 seconds; budget-out instead

## Why this is hard

Computed styles via `getComputedStyle()` always reflect the **current** state of the element. A button at rest returns the resting style; the same button hovered returns the hover style; the same button focused returns the focus style. The browser never returns "all states at once".

The naive approach — parse the stylesheet text and look for `button:hover { ... }` rules — fails badly in modern CSS:

- Most styles live in inline `<style>` blocks generated by tools (Tailwind, CSS-in-JS, Emotion, styled-components) that emit selectors like `.css-a8x7z2:hover` where `a8x7z2` is hash-generated; rule mining requires knowing which hash maps to which element.
- Cascade resolution requires `@media`, `@supports`, `@container`, and specificity reasoning; reimplementing the cascade is roughly the same code budget as writing a browser.
- Tailwind's `hover:` variant produces `.hover\:bg-blue-500:hover` selectors — escape-aware parsing required.
- Cascade Layers (`@layer base, components, utilities`) re-order specificity in ways no plain selector parse can recover.

The robust path is to **let the browser compute it**: position the synthetic pointer over the target element, sample `getComputedStyle()`, move the pointer away, sample again, and diff. The pointer manipulation is a Chrome DevTools Protocol (CDP) input event the dev-browser primitive already exposes. Each state probe takes ~150-300ms; the cascade work is delegated entirely to Chromium.

## Algorithm

### Step 1 — Enumerate stateful candidates

A "stateful candidate" is an interactive element worth probing. The candidate set is fixed per role:

| Role | Selector |
|---|---|
| button | `button, a[role="button"], [type="button"], [type="submit"]` |
| link | `a:not([role="button"])` |
| input | `input:not([type="hidden"]), textarea, select` |
| nav-item | `nav a, nav button` |
| card | `[class*="card"], article` (focusable variant) |

For each role, dev-browser returns up to N landmark instances (`N=3` by default — captures variance without exploding time budget). Each instance gets a stable selector (`button.btn-primary:nth-of-type(2)` style) the probing pass can re-resolve on the live DOM.

### Step 2 — Capture the resting style

Same code path as the default extractor — `getComputedStyle()` over the named computed-style fields (color, backgroundColor, borderColor, borderWidth, boxShadow, outline, outlineColor, transform, opacity, cursor). Stored as the baseline.

### Step 3 — Simulate the state via CDP input + computed-style sample

For each state, the probing sequence is:

| State | How to enter | How to leave |
|---|---|---|
| `:hover` | `Input.dispatchMouseEvent { type: "mouseMoved", x, y }` to the element's center | `Input.dispatchMouseEvent { type: "mouseMoved", x: 0, y: 0 }` to clear |
| `:focus` | `Input.dispatchKeyEvent { type: "rawKeyDown", key: "Tab" }` repeatedly until the element is `document.activeElement`; or `element.focus()` directly via `Runtime.evaluate` | `element.blur()` |
| `:active` | mouseDown but NOT mouseUp at the element's center | mouseUp to release |
| `:disabled` | (NOT simulated — this is a static attribute, not an interaction state; captured by adding `disabled` attribute via `Runtime.evaluate` and sampling, then removing) | remove `disabled` attribute |
| `:focus-visible` | Same as `:focus` BUT the entry must be via keyboard (Tab), not pointer; Chromium only emits `:focus-visible` for keyboard-arrived focus | blur |

After entering each state, wait one animation frame (`requestAnimationFrame` resolution) for transitions / animations to settle to their starting frame — NOT to wait out the full transition; we want the *starting* visual state, not the post-transition state.

Then sample `getComputedStyle()` and store as the state snapshot.

### Step 4 — Restore and diff

After all states are probed, blur every element, move the mouse off-screen (`{ x: -1, y: -1 }`), release any held buttons. The DOM must be returned to its pre-probe state — failing to restore breaks any subsequent extraction on the same page session.

For each `(element, state)` pair, diff the state snapshot against the resting snapshot. Only changed properties are recorded — recording the entire computed style produces 90% duplicate data and bloats DESIGN.md.

## Pseudo-code

```javascript
async function probeStates(devBrowser, role, instances) {
  const probed = []
  for (const inst of instances) {
    const rect = await devBrowser.boundingBoxOf(inst.selector)
    if (!rect) continue

    const resting = await devBrowser.computedStyleOf(inst.selector, FIELDS)

    const hover = await withState("hover", inst, rect, devBrowser, async () => {
      await devBrowser.mouseMove(rect.cx, rect.cy)
      await devBrowser.animationFrame()
      return devBrowser.computedStyleOf(inst.selector, FIELDS)
    })

    const focus = await withState("focus", inst, rect, devBrowser, async () => {
      await devBrowser.evaluate(`document.querySelector(${JSON.stringify(inst.selector)}).focus()`)
      await devBrowser.animationFrame()
      return devBrowser.computedStyleOf(inst.selector, FIELDS)
    })

    const active = await withState("active", inst, rect, devBrowser, async () => {
      await devBrowser.mouseDown(rect.cx, rect.cy)
      await devBrowser.animationFrame()
      const snap = await devBrowser.computedStyleOf(inst.selector, FIELDS)
      await devBrowser.mouseUp(rect.cx, rect.cy)
      return snap
    })

    const disabled = await withState("disabled", inst, rect, devBrowser, async () => {
      await devBrowser.evaluate(
        `document.querySelector(${JSON.stringify(inst.selector)}).setAttribute('disabled','')`
      )
      const snap = await devBrowser.computedStyleOf(inst.selector, FIELDS)
      await devBrowser.evaluate(
        `document.querySelector(${JSON.stringify(inst.selector)}).removeAttribute('disabled')`
      )
      return snap
    })

    probed.push({
      role,
      selector: inst.selector,
      resting,
      states: {
        hover: diff(resting, hover),
        focus: diff(resting, focus),
        active: diff(resting, active),
        disabled: diff(resting, disabled),
      },
    })
  }
  // Cleanup pass — restore DOM to pre-probe state
  await devBrowser.mouseMove(-1, -1)
  await devBrowser.mouseUp(-1, -1)
  return probed
}
```

The `diff(a, b)` function returns `{prop: b[prop]}` for every `prop` where `a[prop] !== b[prop]`. A button whose only state change is `backgroundColor` produces a state record like `{hover: {backgroundColor: "#6f7be0"}}` — concise, easy to consume in the DESIGN.md emitter.

## The state-capture order matters

The probing order is **hover → focus → active → disabled** for one specific reason: each state can leak into the next if not isolated.

- `hover` first because it's the most-likely-no-side-effect state (pointer-only).
- `focus` second because focusing an element while hovering changes nothing (still hovered AND focused — but the probe is for the focus delta, not the hover delta).
- `active` third because mouseDown-without-mouseUp risks accidentally clicking the element and triggering a navigation; the active probe MUST run with `Page.frameNavigated` guarded — abort if a navigation starts.
- `disabled` last because it's a DOM attribute mutation, not an interaction; mutating attributes can re-run framework reconciliation (React, Vue, Lit) and invalidate prior probe state.

After each state, restore BEFORE moving on. The restore sequence is not "move pointer + blur element" once at the end — it is part of each probe's cleanup.

## Bin integration

The `--extract-states` flag of `bin/amw-design-md-from-url.sh` activates this pass:

```bash
bash bin/amw-design-md-from-url.sh https://example.com --extract-states -o DESIGN.md
```

Without the flag, the script runs its default `dev-browser eval` extractor (resting state only, fast path). With the flag, the script:

1. Runs the default extractor first → resting palette + landmark snapshot.
2. Builds the candidate list from the landmark snapshot.
3. Runs the state-probing pass via the dev-browser wrapper (`Input.dispatchMouseEvent`, `Runtime.evaluate`, etc.).
4. Writes both the resting tokens and the state-delta map into the DESIGN.md frontmatter.

The state-probing pass requires the dev-browser wrapper to expose `mouseMove`, `mouseDown`, `mouseUp`, `evaluate`, `animationFrame`, and `computedStyleOf` primitives. If the wrapper lacks any of these, the bin script falls back gracefully with a `warnings: ["dev-browser wrapper lacks state-probe primitives; --extract-states ignored"]` entry and continues with the resting-state extraction only.

## Output schema

```yaml
extensions:
  states:
    button-primary:
      resting:
        backgroundColor: "#5e6ad2"
        color: "#ffffff"
      hover:
        backgroundColor: "#6f7be0"
      focus:
        outline: "2px solid #5e6ad2"
        outlineOffset: "2px"
      active:
        backgroundColor: "#4d59c4"
        transform: "translateY(1px)"
      disabled:
        backgroundColor: "#9ca0b8"
        cursor: "not-allowed"
        opacity: "0.6"
    link:
      resting:
        color: "#5e6ad2"
        textDecoration: "none"
      hover:
        color: "#6f7be0"
        textDecoration: "underline"
      focus:
        outline: "2px solid #5e6ad2"
    input-default:
      resting:
        borderColor: "#3a3d44"
        backgroundColor: "#1a1d23"
      focus:
        borderColor: "#5e6ad2"
        boxShadow: "0 0 0 3px rgba(94, 106, 210, 0.2)"
      disabled:
        backgroundColor: "#26282d"
        cursor: "not-allowed"
```

The `resting` block duplicates the existing `components.<name>` block in the canonical frontmatter — the extension keeps `resting` for convenience so consumers don't need to read two sections to understand a component's state spectrum.

## Breaks if

- **Site uses CSS-in-JS that regenerates class names per render** (React/Emotion in dev mode) — the candidate selectors captured in Step 1 may not match anymore by Step 3. Use stable attribute selectors (`[data-testid]`, `[aria-label]`) when present, or fall back to `:nth-of-type` index-based selectors against a stable parent.
- **The site has motion** — a hover that triggers a 500ms transition will sample mid-transition unless the post-state wait is too long OR too short. The single-animation-frame wait targets the START of the transition (the user-facing "snapped to hover" pixel), not the END. If a site uses `transition-delay`, the snapshot may capture the resting style still; record `warnings: ["state probe may miss transition-delayed states on <selector>"]`.
- **Navigations** — mouseDown on an `<a href="/foo">` can fire navigation on some sites even without mouseUp (browser-dependent). The probe pass MUST guard with a navigation listener and abort the active probe on `Page.frameNavigated` events.
- **Pointer-gated hover** (`@media (hover: hover)`) — Chromium reports `hover: hover` for headless, but headed mode with a touchscreen-emulated viewport reports `hover: none` and the hover styles never apply. Force-set the media-feature override via CDP `Emulation.setEmulatedMedia` before probing.
- **Multiple landmark instances with subtle state differences** — N=3 instances per role may capture three slightly different hover deltas. Reconcile by majority vote on each property; if no majority, record both and flag.
- **Disabled state requires removing `disabled`, not toggling** — some frameworks watch `disabled` for changes and re-render the entire subtree; toggling can break subsequent probes. Always `removeAttribute('disabled')`, never `element.disabled = false`.
