# CLAUDE.md snippet for projects using DESIGN.md

When a project adopts DESIGN.md as its visual source of truth, the project's `CLAUDE.md` (the file Claude Code reads at session start) MUST instruct Claude to read DESIGN.md before generating any UI. The snippet below is what `amw-design-md-author-agent` adds to a project's `CLAUDE.md` (or includes alongside an authored DESIGN.md as a starter).

Adjust the relative path to match where DESIGN.md lives in the project (default: project root).

---

## Snippet — paste into the project's `CLAUDE.md`

```markdown
## Design source of truth — DESIGN.md

This project uses a `DESIGN.md` file as the canonical specification of its visual identity.
Before generating ANY UI (HTML, JSX, CSS, Tailwind classes, component code), read
`./DESIGN.md` end-to-end and follow these rules:

### Hard rules

1. **Tokens are normative.** Color hex values, font sizes, weights, spacing scale
   values, corner radii, and component property tokens defined in the YAML
   frontmatter of DESIGN.md are the *only* permitted values. Never invent new
   colors, sizes, or spacing values without first updating DESIGN.md.

2. **Reference tokens by name in code.** When generating component code, use
   CSS custom properties (`var(--primary)`), Tailwind class names (`bg-primary`),
   or token-mapped utilities — never inline hex values. The companion file
   `./tokens.css` (if present) declares the CSS variable bindings.

3. **Section guidance is binding.**
   - The `## Overview` section states the brand mood and density. Match it.
   - The `## Do's and Don'ts` section lists hard guardrails. Treat these as
     constraints, not preferences.
   - The `## Components` section gives per-component usage rules (e.g.,
     "primary button is used for the single most important action per screen").

4. **If a needed token is missing**, stop and ask. Do not invent values.

5. **Validate before delivery.** When this project's plugin is the
   `ai-maestro-webdesign` plugin, run:
   ```bash
   bash $CLAUDE_PLUGIN_ROOT/bin/amw-design-md-lint.sh DESIGN.md
   ```
   Any errors must be resolved before generated UI is delivered.

### Quick reference (auto-extracted)

When generating code in this project, the most-used tokens are:

- **Primary brand color:** `{{primary-hex}}`
- **Primary surface:** `{{surface-hex}}`
- **Primary text:** `{{text-primary-hex}}`
- **Default font (body):** `{{body-font-family}}` at `{{body-font-size}}`
- **Spacing base:** `{{spacing-base}}` (typically 8px or 16px)
- **Default radius:** `{{radius-default}}`

(These mirror DESIGN.md frontmatter. If they drift, DESIGN.md wins.)

### File index

- `./DESIGN.md` — the design system spec
- `./tokens.css` — CSS custom properties (auto-emitted)
- `./tokens.json` — W3C Design Tokens format (auto-emitted)
- `./component-inventory.md` — list of declared components and their token slots

These four files are regenerated by `bin/amw-design-md-emit-companions.py` whenever
DESIGN.md changes. Do not edit them by hand; edit DESIGN.md and re-run the emitter.
```

## Where this goes in CLAUDE.md

If the project already has a `CLAUDE.md`:
- Add the snippet under a new `## Design source of truth — DESIGN.md` section, near the top, after any "what this repo is" section.
- Update the existing CLAUDE.md sections that reference styling/UI to defer to DESIGN.md.

If the project does NOT have a `CLAUDE.md`:
- Create one with the snippet as its first content block.
- Add a one-line "What this repo is" section above the snippet.

---

## What `{{placeholders}}` get filled with

The author agent fills `{{primary-hex}}`, `{{surface-hex}}`, etc. by reading the just-authored DESIGN.md frontmatter. If a token is absent (e.g., the user did not declare a `surface` token), the placeholder is removed from the snippet entirely — never left as `{{...}}`.
